# Execution Model — How the OS Turns Strategy Into Action

This file is the decision framework for operationalizing any plan in Spyglass OS. Read it when working through a strategic plan (marketing plan phases, product roadmap, launch checklist) and deciding how each item should be tracked and executed.

## The three-tool rule

Every piece of work belongs to exactly one of three places:

| Tool | Job | What goes here |
|---|---|---|
| **Linear** | Tasks with owners and deadlines | One-time setup items, project sprints, anything with a clear "done" state, assignments between Enitan and Teresa |
| **OS skills + rituals** | Execution and intelligence | Recurring execution (write, post, outreach), cadence-driven review, parameter adjustment, signal scoring |
| **Wiki** | Durable knowledge | Architecture decisions, how things work, why choices were made, synthesized research |

**The rule:** If you find yourself trying to track a task inside a ritual, or trying to explain a recurring process inside a Linear issue, you have it in the wrong place.

## The decision test — Linear or ritual?

Ask these questions in order. Stop at the first yes.

**Send to Linear if:**
1. Is it a one-time task? (setup, configuration, build, draft a specific thing once)
2. Does it have a specific owner — Enitan or Teresa?
3. Does it have a deadline that would cause a problem if missed?
4. Does it depend on another task completing first?
5. Does it need to be visible to Teresa without her running the OS?

**Send to a ritual or skill if:**
1. Will it happen more than once a month, indefinitely?
2. Is the output always the same type (article, post, signal report, parameter review)?
3. Does the timing depend on a cadence (weekly, monthly, exam calendar) rather than a deadline?
4. Is the action generated by data (PostHog, GSC, Postiz) rather than a plan?

**Send to the wiki if:**
1. Is it knowledge that should outlast the current project phase?
2. Is it a decision with reasoning that future sessions should understand?
3. Is it reference material that multiple skills need to load?

## Phase-based application

This framework applies differently depending on where you are in a plan.

**Project phases (Phase 0, new channel setup, product launches):**
Primarily Linear. Most work is one-time setup with owners and deadlines. The OS queries Linear via MCP at session start to surface open items. Once setup is complete, Linear fades.

**Steady-state execution (Phase 1+, ongoing channels):**
Primarily OS rituals and skills. Content calendar drives writing. Weekly pulse drives review. Tune drives adjustment. Linear becomes secondary — only used for significant new work items.

**Phase transitions:**
The `/phase-gate` skill (to be built) scores gate criteria, outputs pass/fail, and drafts a `decisions/log.md` entry. The transition itself is a decision — it goes to the log, not to Linear.

## What this is NOT

- Not a task management methodology. The OS doesn't manage tasks — Linear does.
- Not a replacement for `continuous-improvement.md`. That governs how repeatable functions get better over time. This governs where work is tracked.
- Not static. If a project phase drags on and Linear items become recurring, they should be converted to rituals. If a ritual generates one-time decisions, those decisions go to the log.
